# DevDocs AI

A full-stack RAG (Retrieval-Augmented Generation) platform that lets you upload a PDF and ask natural language questions about it — with every answer grounded in the actual document content and backed by source citations.

**Live demo:** [https://devdocs-ai-sizy.onrender.com](https://devdocs-ai-sizy.onrender.com) *(backend)* · Frontend: *[https://devdocs-ai-iota.vercel.app/](https://devdocs-ai-iota.vercel.app/)*

---

## What it does

1. Upload any PDF document
2. The system extracts, chunks, and embeds the text into a vector database
3. Ask questions in plain English
4. Get answers generated by an LLM — grounded only in your document, with source excerpts shown for every answer

No hallucinated answers. If the document doesn't contain the answer, the system says so.

---

## Tech Stack

**Backend**
- **FastAPI** — REST API framework
- **LangChain** — document loading, chunking, and prompt orchestration
- **ChromaDB** — vector database for semantic search, with isolated per-document collections
- **Google Gemini API** — `gemini-embedding-001` for embeddings, `gemini-2.5-flash` for answer generation
- **Pydantic** — request/response validation
- **PyPDF** — PDF text extraction

**Frontend**
- **React** (TypeScript)
- **Tailwind CSS**
- **Axios** for API communication

**Deployment**
- Backend: **Render**
- Frontend: **Vercel**
- Both connected via GitHub for continuous deployment

## How it works (architecture)

```
PDF Upload
    ↓
Text Extraction (PyPDF)
    ↓
Chunking (LangChain — 1500 chars, 150 char overlap)
    ↓
Embedding Generation (Gemini Embeddings)
    ↓
Vector Storage (ChromaDB)
    ↓
User Question → Embedded → Semantic Search (top 6 chunks)
    ↓
Context + Question → Gemini 2.5 Flash
    ↓
Grounded Answer + Source Citations
```

**Chunking strategy:** Chunk size was tuned from 500 → 1500 characters after observing that smaller chunks fragmented project/section context across boundaries, causing incomplete retrieval. Larger chunks with overlap kept semantically related content intact.

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/` | Health check |
| `POST` | `/upload` | Upload a PDF, returns a `doc_id` |
| `POST` | `/query` | Ask a question about an uploaded document |
| `GET` | `/documents` | List all uploaded documents |

Interactive API docs available at `/docs` (auto-generated via FastAPI/Swagger).

## Running locally

### Backend

```bash
cd backend
python -m venv venv
venv\Scripts\activate        # Windows
pip install -r requirements.txt
```

Create a `.env` file in `backend/`:

```
GOOGLE_API_KEY=your-gemini-api-key
```

Run the server:

```bash
uvicorn main:app --reload
```

### Frontend

```bash
cd frontend
npm install
npm start
```

Update `API_URL` in `src/App.tsx` to point to your backend if not running locally on port 8000.

## Key engineering decisions

- **Per-document vector collections** — each uploaded PDF gets its own ChromaDB collection, preventing cross-document contamination during retrieval
- **Source citation on every answer** — retrieved chunks are returned alongside the answer for traceability and to reduce hallucination risk
- **Grounded prompting** — the system prompt explicitly instructs the model to say "I don't know" rather than guess when context is insufficient
